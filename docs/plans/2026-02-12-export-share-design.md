# 导出与分享功能 — 决策记录

> 2026-02-12 brainstorming session

## 背景

4 个模板（Shadcn / Claude / Terminal / Editorial）已完成，输出方式只有 PDF（window.print）。讨论是否拓展输出途径。

---

## Q1: 项目的部署约束是什么？

这会直接决定哪些输出方式可行 — 纯前端能做的事和有后端能做的事差别很大。

**决定：不设限，先看方案再决定。**

---

## Q2: 除了 PDF，你在想哪些输出形式？

选项：在线链接 / 图片导出 / 纯文本复制

**决定：在线链接。** 核心诉求是"发给面试官一个链接就能看"，比 PDF 附件更轻量。

---

## Q3: "在线链接"具体是哪种模式？

三个方向：

1. **纯前端编码链接** — 数据压缩进 URL hash，零后端，但 URL 可能很长
2. **服务端存储 + 短链接** — 数据存服务器，生成短 URL，类似 resume.io
3. **导出独立 HTML 文件** — 用户自己部署到 GitHub Pages / Vercel / Netlify

**推荐导出 HTML 的理由：**
- 零后端成本，不需要数据库、不需要处理链接过期
- 用户完全掌控文件，想放哪放哪
- 技术上最自然 — 模板本身就是 HTML+CSS，打包即可
- 和现有架构完美契合 — 从 previewRef 取 HTML 包装成完整文档
- 可叠加 — 后续做"托管短链接"本质上也是把这个 HTML 托管到某处

**否决的方案：**
- 纯前端编码链接：数据量大了 URL 会爆，简历内容暴露在 URL 里有隐私问题
- 服务端短链接：需要后端、数据库、用户认证，复杂度跳了一个量级

---

## Q4: 但是有没有办法让用户直接拿到链接？

用户反馈：HTML 导出 OK，但还是希望能更直接 — 点一下就拿到可分享的链接。

重新审视"纯前端编码链接"方案：
- lz-string 压缩后，典型简历 JSON（1-3KB）可以压到 500B-1.5KB
- 现代浏览器支持 64KB+ 的 URL，实际可行
- 数据在 hash 里（`#` 后面），不会发送到服务器，隐私问题可控

**决定：两个都做。**
- HTML 导出作为"正式部署"路径
- 压缩链接作为"快速分享"路径
- 两条路互补，不冲突

---

## Q5: 别人打开分享链接时看到什么？

选项：
1. **纯净查看模式** — 只有简历，底部一个小 "Powered by" 链接
2. **只读编辑器模式** — 能切模板、下载 PDF，但不能编辑

**决定：纯净查看模式。**
理由：发给面试官的页面应该干净专业，不需要编辑器 UI 干扰。底部的 "md-resume" 链接既是品牌露出，也是获客入口。

---

## 最终方案

| 功能 | 实现方式 | 用途 |
|------|---------|------|
| 导出 HTML | previewRef.innerHTML → Blob 下载 | 正式部署、离线查看 |
| 分享链接 | lz-string 压缩 → URL hash → 剪贴板 | 快速分享、即时预览 |
| 查看模式 | App.tsx hash 路由检测 → ViewMode 组件 | 分享链接的渲染页面 |

新增依赖：`lz-string`（~4KB gzipped）

文件变更：
- `src/App.tsx` — hash 检测、三个导出按钮、toast
- `src/components/ViewMode.tsx` — 纯净查看模式
- `src/utils/export-html.ts` — HTML 导出
- `src/utils/share.ts` — 压缩/解压/URL 生成
